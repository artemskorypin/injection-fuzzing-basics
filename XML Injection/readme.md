# XXE Injection wordlists

Этот раздел — про **XML External Entity (XXE)** инъекции.

Фокус тот же, что и в остальной части репозитория:

- **детект и фаззинг**, а не максимальная эксплуатация;
- **одна строка = один payload**;
- минимум дубликатов, каждая строка — отдельная идея (тип источника, протокол, контекст).

Сейчас в разделе используется один файл:

---

## Файл: `XML_XXE.txt`

Содержит небольшой, но разнообразный набор XXE-нагрузок:

- **Классический inline XXE (file/http)**  
  DOCTYPE + ENTITY + подстановка сущности в тело документа:
  - чтение локальных файлов (`file:///etc/passwd`, `/etc/issue`, `c:/boot.ini`);
  - обращение к внешнему HTTP-ресурсу (`http://example.com/xxe`).

- **DTD/ENTITY-фрагменты и обёртки**  
  Примеры использования:
  - `php://filter/convert.base64-encode/resource=/etc/passwd` и `index.php` для base64-дампа содержимого;
  - `data://` как источник (inline-данные вместо файлов/URL);
  - параметр-энтити (`%xxe`, `%dtd;`) и внешние DTD (`http://example.com/evil.dtd`).

- **XXE в SOAP / CDATA**  
  Пример, когда вредоносный DOCTYPE/DTD прячется внутри `<![CDATA[ ... ]]>` в SOAP-сообщении.

- **XInclude**  
  Вариант XXE без DOCTYPE, через:
  - `xmlns:xi="http://www.w3.org/2001/XInclude"`
  - `<xi:include parse="text" href="file:///etc/passwd"/>`

- **XXE → SSRF**  
  Нагрузки, где XML-парсер ходит по HTTP:
  - на `127.0.0.1` (локальные сервисы),
  - к метадате облака (`169.254.169.254/latest/meta-data/...`),
  - к произвольному внутреннему сервису.

- **Blind / Out-of-Band XXE**  
  Примеры, которые не зависят от содержания ответа приложения:
  - загрузка внешней DTD/ENTITY с вашего домена (`YOUR-COLLABORATOR-ID.example.com`);
  - попытка приклеить содержимое файла (`/etc/passwd`) к HTTP-запросу и увидеть его хотя бы частично в логах.

## Про DoS / Billion Laughs

В словарь **специально не включены** payload’ы для DoS-атак через XXE, например:

- бесконечные/экспоненциальные сущности (Billion Laughs),
- чтение специальных устройств вроде `file:////dev/random`.

Такие конструкции относятся уже к **явной эксплуатации и отказу в обслуживании**, а не к базовому фаззингу/детекту, и поэтому:

- их стоит использовать только в изолированных/тестовых средах,
- при необходимости их проще добавить вручную под конкретный сценарий, чем держать в общем словаре.

## Использование

- файл `XML_XXE.txt` можно скормить как **payload-лист** для параметров, принимающих XML (тело запроса, части SOAP, параметры с XML-фрагментами);
- каждая строка — готовый пример, который можно:
  - отправлять целиком (если приложение ожидает сырое XML-тело),
  - или использовать как шаблон и адаптировать под конкретный формат/корневой элемент.
